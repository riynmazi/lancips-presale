Overview

Dokumen ini menjelaskan alur utama aplikasi: bagaimana data diambil dari sumber (mis. API DexScreener), diproses di backend, lalu dirender di frontend.

  • Data source: API pihak ketiga atau database internal.
  • Backend: service yang melakukan fetch, normalisasi, caching, dan scoring.
  • Frontend: menampilkan card, filter, dan visualisasi; re-render saat data baru masuk.

Arsitektur

Ringkasan komponen dan tanggung jawabnya.

  1. Fetcher — cron job / worker yang memanggil API eksternal setiap interval (mis. 30s–1m) lalu menyimpan hasil ke cache/DB.
  2. Processor — menghitung metrik (volume, liquidity, score viral), menormalisasi field, dan menyiapkan payload untuk frontend.
  3. API / Gateway — endpoint publik/privat untuk frontend mengambil data (REST / GraphQL / websocket untuk realtime).
  4. Frontend — UI yang mengambil data, mendukung filter, search, dan rendering card. Gunakan debounce pada search dan pagination/virtualization untuk performa.

Tip performa: aktifkan caching (Redis), gunakan worker queue (Bull, RabbitMQ), dan batasi payload yang dikirim ke frontend.

API — contoh endpoint

Contoh minimal endpoint yang diharapkan frontend panggil.

GET /api/v1/memes?chain=bsc&page=1&limit=20

Response: { "data": [ {"symbol":"DOGE","name":"Doge","volume_24h":136000,"liquidity":12000,"score":87,"meta":{...}}, ... ], "meta": {"page":1,"limit":20,"total":124} }

Jika butuh realtime: sediakan WebSocket / SSE yang mengirimkan perubahan (patch) agar frontend cukup apply diffs dan tidak harus re-fetch seluruh halaman.

FAQ & Troubleshoot

MasalahSolusi singkat
Data tidak munculPeriksa CORS, console error, dan endpoint API (status 200). Pastikan cache tidak mengembalikan stale response.
Render lambatGunakan pagination, lazy-loading gambar, dan virtual scroll.
Score viral anehPeriksa formula scoring—pastikan normalisasi (z-score atau min-max) berlaku terhadap window waktu yang konsisten.

  • Caching (Redis)
  • Observability (logs, metrics)
  • Security (CORS, auth)
